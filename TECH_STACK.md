# 기술 스택 선정 근거 및 대안 비교

> 인천 1인가구·세대구조 정책 대시보드. **소규모(시군구 10개) 정적 데이터 · 서버 없음 ·
> 지도 중심 인터랙티브 시각화 · 단일 HTML로 배포**라는 요구가 스택 선택의 기준이었다.
> 아래는 각 계층에서 왜 이 기술을 골랐고, 대안 대비 무엇이 유리/불리한지 정리한 것이다.

---

## 요약 표

| 계층 | 채택 | 주요 대안 | 한 줄 선정 이유 |
|---|---|---|---|
| 프론트엔드 | **React** | Vue, Svelte | 지도·차트 라이브러리 생태계와 상태 공유(연동 뷰) 성숙도 |
| 빌드 도구 | **Vite** | CRA(폐지), webpack, Parcel | 빠른 개발 + 단일 HTML 빌드 플러그인 |
| 지도 | **Leaflet (react-leaflet)** | MapLibre GL, OpenLayers | 폴리곤 10개 규모엔 가장 가볍고 단순 |
| 배경지도 | **OSM/CARTO Positron** | 순수 OSM, Mapbox, VWorld | 무료·키 불필요·중립 톤(데이터 색과 안 싸움) |
| 차트 | **Recharts** | Chart.js, D3, ECharts, visx | React 선언형·표준 차트에 개발속도 최고 |
| 경계 전처리 | **pyshp+pyproj(+shapely)** | geopandas/GDAL | GDAL 없이 pip 설치만으로 SHP→GeoJSON |
| 배포 형태 | **단일 HTML(singlefile)** | 다중 파일 정적 호스팅 | "파일 하나 열면 끝" 요구 충족 |
| 데이터 | **빌드타임 정적 JSON** | 실시간 API/DB | 공개·저빈도 갱신 데이터 → 서버 불필요 |

## 1. 프론트엔드 — React (vs Vue, Svelte)

**선택 이유.** 이 대시보드의 핵심은 "지도·순위·산점도·상세 패널이 하나의 선택 상태(선택 시군구,
지표, 유형 필터)를 공유하며 서로 연동"되는 구조다. 이런 **다중 연동 뷰(linked views)** 는
컴포넌트 + 공유 상태(hooks) 모델에 잘 맞고, React는 이를 위한 생태계가 가장 두껍다.
특히 지도(`react-leaflet`)와 차트(`recharts`)의 **React 바인딩이 성숙**해, 지도·차트를
같은 상태에 연결하는 배선 비용이 작다.

**대안 비교.**
- **Vue** — 학습 곡선이 완만하고 템플릿 문법이 직관적이다. `vue-leaflet`·차트 래퍼도 있으나
  성숙도·레퍼런스가 React보다 얕다. 기능적으로 못 할 이유는 없지만, 지도+차트 조합의
  "바로 쓰는" 컴포넌트가 React 쪽이 더 많아 개발 속도에서 앞선다.
- **Svelte** — 컴파일 방식이라 런타임이 얇고 번들이 작으며 반응성 문법이 간결하다. 단일 HTML
  경량화엔 매력적이다. 그러나 `svelte-leaflet` 등 지도·차트 바인딩 생태계가 얇아, 지도 팝업·
  양방향 연동을 직접 구현하는 부담이 크다. 시군구 10개 규모에서 Svelte의 런타임 성능 이점은
  체감되지 않으므로, **이점(성능·번들)보다 생태계 손실이 더 크다**고 판단.

**결론.** 규모가 작아 런타임 성능은 변별력이 없고, 변별 요소는 **라이브러리 상호운용성**이다 →
React가 유지보수·확장·인력 측면에서 가장 안전.

## 2. 빌드 도구 — Vite (vs CRA, webpack, Parcel)

**선택 이유.** 네이티브 ESM + esbuild 기반으로 개발 서버가 즉시 뜨고 설정이 사실상 없다.
`@vitejs/plugin-react`로 React가 바로 붙고, 결정적으로 **`vite-plugin-singlefile`** 로
JS/CSS/데이터를 `index.html` 하나에 인라인해 "파일 열기만으로 실행" 요구를 충족한다.

**대안 비교.**
- **CRA(Create React App)** — 공식적으로 유지보수 중단. 개발 서버가 느리고, 세밀한 설정을
  하려면 eject 해야 해 관리 부담이 크다.
- **webpack** — 강력하고 유연하지만 설정이 장황하다. 이 규모에선 과한 도구.
- **Parcel** — 무설정 철학은 Vite와 유사하나, React 생태계 표준·플러그인(특히 singlefile류)
  선택지는 Vite가 더 풍부.

**결론.** 현시점 React SPA의 사실상 표준이자, 단일 HTML 산출 요구까지 플러그인으로 해결 → Vite.

## 3. 지도 — Leaflet / react-leaflet (vs MapLibre GL, OpenLayers)

**선택 이유.** 표현 대상은 **시군구 10개 폴리곤 + 배경 타일 + 색상 choropleth + 클릭/호버/팝업**
정도다. Leaflet은 ~40KB로 가볍고 API가 단순하며 래스터 타일·GeoJSON 오버레이·팝업이 기본 제공된다.
`react-leaflet` 바인딩이 안정적이라 선택 상태와의 연동도 수월하다. **요구 대비 가장 적정한 무게.**

**대안 비교.**
- **MapLibre GL JS** — WebGL 벡터 타일로 부드러운 줌·데이터 기반 스타일링·대용량/3D에 강하다.
  그러나 벡터 타일 소스와 스타일 JSON을 갖춰야 하고 구조가 무겁다. **폴리곤 10개엔 이점이
  발현되지 않고 복잡도만 늘어난다.** (전국 읍면동처럼 수천~수만 피처, 벡터 스타일링, 고성능
  렌더가 필요해지면 그때 MapLibre가 정답.)
- **OpenLayers** — 투영(EPSG) 변환, WMS/WFS, 편집 등 본격 GIS 기능이 강력해 공공 GIS에서 많이
  쓰인다. 대신 API가 방대하고 장황하며 번들도 크다. 이 대시보드는 좌표계를 전처리 단계(pyproj)에서
  이미 4326으로 맞추고 WMS·편집도 불필요하므로 **OpenLayers의 강점이 쓰일 데가 없다.**

**결론.** "소수 폴리곤 choropleth + 기본 인터랙션"에는 Leaflet이 무게·복잡도 균형에서 최적.
데이터가 읍면동·전국 단위로 커지거나 벡터 스타일 성능이 필요해지면 **MapLibre로 이관**하는 것이
합리적 확장 경로다.

## 4. 배경지도 — OSM 데이터 / CARTO Positron 타일 (vs 순수 OSM, Mapbox, VWorld)

**선택 이유.** choropleth는 배경이 튀면 데이터 색과 충돌한다. CARTO **Positron(light_all)** 은
회백색의 중립 톤이라 색칠된 시군구가 위에서 잘 읽히고, **무료·API 키 불필요**로 배포 부담이 없다.
타일의 원 데이터는 OSM이라 지명·해안선이 정확하다.

**대안 비교.**
- **순수 OSM 표준 타일** — 색이 화려해 데이터 위에 깔면 시끄럽고, OSM 타일 서버는 대량/상용
  사용을 정책적으로 권장하지 않는다.
- **Mapbox** — 품질은 뛰어나나 액세스 토큰·과금이 필요해 "그냥 열면 되는" 공유엔 부담.
- **VWorld(국토부)** — 국내 공식·한글 라벨이 강점이지만 API 키와 도메인/CORS 제약이 있어 정적
  단일 HTML 배포와 궁합이 덜 맞는다.

**결론.** 키 없이 무료로 쓰는 중립 배경 → CARTO Positron. **국토연 등 공식 산출물에서 국내 공식
배경도가 요구되면 VWorld로 교체**(키 발급·도메인 등록 후 TileLayer URL만 변경)하면 된다.

## 5. 차트 — Recharts (vs Chart.js, D3, ECharts, visx)

**선택 이유.** 필요한 차트는 막대(순위)·100% 누적막대(연령구조)·산점도(사분면)·라인(추이)·
도넛(구성) 등 **표준형**이다. Recharts는 React 컴포넌트로 SVG 차트를 선언적으로 그려, 선택/호버
상태를 props로 넘겨 **지도와의 연동을 그대로 배선**할 수 있고 기본 스타일도 준수하다.

**대안 비교.**
- **Chart.js** — Canvas 기반으로 대용량에 빠르나 명령형 API라 React 상태·개별 요소 클릭 연동이
  번거롭다. SVG가 아니라 개별 도형 하이라이트도 덜 유연.
- **D3** — 궁극의 유연성이지만 저수준이라 같은 결과를 훨씬 많은 코드로 만든다. 맞춤형 비표준
  시각화가 필요할 때 값어치가 있다.
- **ECharts** — 기능이 매우 풍부하고 대용량·복합 차트에 강하나 무겁고 React 관용구와는 거리가 있다.
- **visx** — D3 + React 프리미티브로 유연하지만 조립 비용이 크다.

**결론.** 표준 차트 + 적은 데이터 + React 상태 연동이 핵심 → Recharts가 개발 속도·연동성에서 최적.
비표준·초대용량·정교한 커스텀이 필요해지면 D3/visx(유연성) 또는 ECharts(대용량)로 이동.

## 6. 경계 전처리 — pyshp + pyproj + shapely (vs geopandas/GDAL)

**선택 이유.** 원본은 국토지리정보원 연속수치지형도 SHP(EPSG:5186)다. 이를 인천만 필터 →
**시군구 단위로 병합(디졸브)** → 4326으로 재투영 → GeoJSON으로 만든다. `pyshp`·`pyproj`·
`shapely`는 **모두 순수 파이썬 휠(wheel)로 제공**되어 macOS(M1/M2 포함)에서 `pip install`만으로
끝난다 — 검토자가 받아 바로 돌릴 수 있는 게 중요했다.

**대안 비교.**
- **geopandas** — `read_file`/`dissolve`/`to_crs` 한두 줄로 끝나는 편의성은 최고지만, 내부적으로
  **GDAL/fiona** 네이티브 의존을 끌어와 설치가 무겁고 환경에 따라 실패가 잦다. (휠이 좋아졌지만
  여전히 리스크.) 그래서 **선택적 폴백**으로만 두고, 기본은 GDAL이 필요 없는 조합을 택했다.

**결론.** "리뷰어 노트북에서 한 번에 실행"을 최우선하면 네이티브 의존(GDAL) 최소화가 정답 →
pyshp/pyproj/shapely. 이미 geopandas가 깔린 환경이면 자동으로 그 경로를 쓴다.

## 7. 배포 형태 — 단일 HTML(vite-plugin-singlefile)

**선택 이유.** 상대가 리액트 프로젝트를 빌드·실행하지 않고 **파일 하나를 더블클릭**하거나 GitHub
Pages 링크로 바로 보게 하는 것이 요구였다. singlefile 플러그인이 JS/CSS/데이터/경계를 `index.html`
하나에 인라인해 `file://`에서도 열린다.

**대안 비교.** 다중 파일 정적 번들은 상대 경로·서버가 필요해 "파일 하나" 요구를 못 채운다.
GitHub Pages로 다중 파일을 올려도 되지만, **단일 파일이 이동·첨부·아카이브에 가장 단순**하다.

---

## 8. 데이터 파이프라인 — 빌드타임 정적 JSON (vs 실시간 API/DB)

**선택 이유.** 데이터는 **공개 통계 · 저빈도 갱신**(인구총조사 연 단위, 주민등록 월 단위)이다.
`extract.py`가 원본 CSV/SHP를 정제·집계·유형화해 정적 JSON/GeoJSON으로 미리 굳히면, 서버·DB
없이 무료로 호스팅되고 로딩도 빠르다.

**대안 비교.** 실시간·대용량·질의형 데이터라면 API/DB가 필요하지만, 여기선 인프라·비용·운영 부담만
늘어날 뿐 이점이 없다. 갱신은 "새 CSV 넣고 `extract.py` 재실행 → 재빌드"로 충분.

## 확장 시 갈아탈 지점 (트리거)

- 피처가 **읍면동·전국 단위**로 커지거나 벡터 스타일/고성능 렌더가 필요 → **MapLibre GL**
- **비표준·정교한 커스텀 시각화**가 필요 → D3/visx, 대용량 복합 차트 → ECharts
- **실시간·질의형 데이터**가 필요 → 백엔드 API + DB 도입
- **국내 공식 배경도**가 필수 → VWorld 타일로 교체(키·도메인 등록)
- 번들 크기가 병목이 되면 → Svelte 재작성 검토(현재 규모에선 불필요)
